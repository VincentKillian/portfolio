Maitriser les bases de Git/GitHub

Mercredi, 01 Fevrier 2023

Ton project est en panne ! Que se passe-t-il cette fois-ci ? Les scénarios de test que tu as écrits couvraient tous les cas d'utilisation que tu pouvais imaginer - et ils y sont tous passé après que tu es mis à jour ton pc hier. Si seulement on pouvais revenir en arrière et examiner toutes les modifications récentes du code pour voir ce qui n'a pas fonctionné ! C'est le type de problème que Git doit résoudre..

Dans cet article, je vais te montrer comment utiliser Git pour suivre l'historique des fichiers de ton projet afin que tu puisses identifier les bogues chaque fois qu'ils apparaissent dans ton projet.

Quelle est la différence entre Git et GitHub ?

GitHub est une application web qui permet aux utilisateurs d'héberger des fichiers, un peu comme DropBox ou Google Drive, mais elle est conçue pour héberger des dépôts.
Git est un outil en ligne de commande utilisé principalement par les programmeurs pour gérer les versions (l'historique) des projets logiciels. Chaque projet vit dans un dépôt Git qui conserve l'historique de toutes les modifications.

1. Installation de Git.

En cliquant ici, tu peux installer Git en cliquant sur le système d'exploitation que tu souhaite utilisez. Après avoir installé Git, vas dans ton invite de commande si tu es sous Windows ou dans le terminal si tu es sous MacOS ou Linux. Tape git et appuis sur la touche Entrée. Si tu vois des informations sur Git, C'est installé correctement ! Si tu obtiens command not found - essaye de désinstaller et de réinstaller git, ou de reconfigurer les variables d'environnement.

2. Créer un compte GitHub

En cliquant ici, ça te mènera à GitHub. De là, tu peux créer un compte en utilisant ton adresse e-mail.

3. Créer votre premier référentiel

Sur n'importe quelle page de GitHub, lorsque tu es connecté, tu verras un signe plus blanc à côté de ton image de profil dans la barre d'outils en haut à droite. En cliquant sur le signe plus, tu verras apparaître un menu déroulant dans lequel tu pourras sélectionner : Nouveau répertoire. À partir de là, tu dois nommer le dépôt de quelque chose comme Maitrise-les-bases-sur-les-G - en tenant compte du fait qu'il ne te permet pas d'utiliser des espaces dans le nom. Tu n'as pas besoin d'ajouter une description et tu peux laisser l'accès par défaut Public activé.
Pour cette 'leçon', tu ne dois pas cocher la case Initialiser avec le fichier Readme, et enfin tu n'as pas besoin d'ajouter un fichier .gitignore ou une licence - tu peux donc cliquer sur le bouton Créer un dépôt.

La page suivante contiendra beaucoup de petits textes un peu intimidants. La seule information dont tu dois te préoccuper sur cette page est la case bleue en haut de la page, intitulée Configuration rapide. Dans la case bleue Configuration rapide, assure-toi que HTTPS est sélectionné et non SSH. À l'extrême gauche du menu, il y a une icône de presse-papiers que tu peux sélectionner pour copier l'adresse URL dont tu as besoin pour cloner ton dépôt sur ton pc.

Dans ton terminal ou git bash, vas dans le répertoire dans lequel tu souhaite placer ton dépôt en utilisant la commande cd nom_répertoire. Exécute ensuite la commande suivante (remplace mon URL par l'URL que tu as copiée de GitHub) : 

git clone https://github.com/VincentKillian/master-the-basics-of-git-github.git
Clonage du référentiel et ensuite changement de répertoire dans le référentiel

L'exécution de cette commande doit créer un dossier vide dans le répertoire à partir duquel tu as exécuté la commande. Tu peux considérer le clonage du repertoire comme son téléchargement, mais en plus grand. Il télécharge également l'historique de toutes les modifications apportées à ces fichiers. Exécute ensuite la commande cd Master_Git_And_GitHub_Basics_Article dans ton invite de commande ou terminal pour aller dans ton nouveau dépôt que tu viens de cloner. Maintenant que tu es dans celui-ci, tu dois pouvoir obtenir le statut de ton repertoire en utilisant la commande : 

git status

La commande git status te montrera toutes les modifications que tu as apportées à ton dépôt depuis ton dernier commit. Elle dira :

Vérification de l'état de notre dépôt en utilisant "git status"

Tout ce que cela nous dit, c'est que tu n'as pas encore d'historique parce que n'as pas encore de commits. Tu peux te demander ce que signifie On branch main - mais je vais aborder ce sujet dans le prochain article. Maintenant, tu vas ajouter de l'historique.

4. Créer notre premier commit

Dans un dépôt Git, on enregistre l'historique en commits. Comme on l'a vu dans la sortie git status, tu n'avais actuellement aucun commit. Afin de créer ton premier commit, tu dois faire quelques changements dans ton dépôt. Crée un nouveau fichier dans ton explorateur de fichiers ou dans la fenêtre de ton finder et fais-le glisser dans le répertoire de ton dépôt. Le fichier sera simplement file.txt sans contenu. Maintenant, si tu retourne à ton invite de commande ou à ton terminal et que tu le lance git status, tu verras ton nouveau fichier :

Utilisation de la commande "touch" pour créer un fichier et exécution de "git status" pour voir les modifications apportées au dépôt

Maintenant tu peux voir à quel point git status est utile pour nous montrer tes changements. Tu remarqueras que le fichier file.txt est listé dans les fichiers non suivis. Rappele-toi que Git veut suivre tous les changements qui se produisent dans le dépôt - mais tu dois lui dire quels fichiers suivre. Pour rendre cela apparent, ajoute un second fichier appelé file2.txt afin de voir ce qui se passe lorsque tu suis un fichier mais pas l'autre. Après avoir ajouté un second fichier au dépôt et vérifié qu'il apparaît comme non suivi lorsque tu execute git status, tu vas apprendre ta prochaine commande Git : 

git add file.txt

Tu selectionneras les fichiers à suivre en utilisant git add nom_fichier. Notez que tu n'as sélectionné que le premier fichier. Maintenant, lorsque tu exécute git status, tu obtiens la sortie :

Créer file2.txt, ajouter file.txt pour qu'il soit prêt à commiter, et vérifier le statut du dépôt

Remarque-tu que file.txt est listé sous un nouvel en-tête : Changes to be committed et que file2.txt est toujours listé sous Untracked. Garde à l'esprit que tu n'as pas suivi quoi que ce soit en ajoutant le fichier - tu viens de sélectionner le fichier à suivre lors de la validation . Maintenant que tu as sélectionné les changements que tu as ajouter à notre commit, tu peux créer ton premier commit dans ton projet en utilisant : 

git commit -m "add file.txt"
Validation du fichier.txt que nous avons ajouté à l'étape précédente

Cela t'indique que tu as ajouté un nouveau commit qui est le root-commit puisque c'est le premier. Les futurs commits ne seront pas des root commits. Ton commit a le message add file.txt, et il n'inclut qu'un seul changement de fichier où tu as créé le file.txt. Enfin, tu apprendra une autre commande qui répertoriera tous les commits de ton repertoire par ordre décroissant : 

git log

Cette commande affiche :

Utilisation de git log pour voir tous nos commits précédents

Un seul commit est répertorié puisque tu n'en n'as qu'un et il nous indique également l'auteur, la date et le message pour chaque commit.

5. Pousser vos modifications

Une fois que tu as validé tes modifications, personne ne pourra les voir sur ton profil GitHub tant que tu n'auras pas envoyé ton code avec la commande : 

git push
Pousser vos modifications de votre machine locale vers votre référentiel GitHub

Tes modifications ont maintenant été téléchargées sur ton GitHub pour que tout le monde puisse les voir. Vas vérifier !

Voir le fichier file.txt que nous avons poussé de notre référentiel local vers notre référentiel en ligne GitHub

Remarques-tu que file2.txt n'est pas là puisque tu ne l'as pas sélectionné pour être commité. Tu peux cliquer sur 1 commit à l'extrême gauche de la barre d'outils au milieu de l'écran pour voir tous les commits un peu comme la façon dont git log te le montre.

6. Et maintenant ?

Tu sais maintenant comment utiliser les commandes git : clone , status , add , commit , log et push . Tu peux maintenant continuer à mettre à jour tes repertoire avec de nouvelles modifications. git ajoutez les fichiers que tu modifie pour les sélectionner à valider . Ensuite, engage-les avec un message décrivant ce que tu as fait et pousse- les vers GitHub pour que les autres puissent les voir. Pour le prochain tutoriel, On apprendras tout sur l'utilisation des branches git pour collaborer avec d'autres développeurs sur des projets logiciels.
